# Integration Consistency - has_integration Field

## Overview

The `has_integration` field in the `institutions` table indicates whether an institution has a programmatic API integration implementation in the codebase. This document explains the logic and ensures consistency between the database and code.

## Current Integration Implementations

The following integrations are implemented in `packages/integrations/src/implementations/`:

1. **BinanceIntegration** - For Binance crypto exchange
2. **BitcoinIntegration** - For Bitcoin blockchain
3. **EvmChainIntegration** - For EVM-compatible blockchains (Ethereum, BSC, Polygon, etc.)
4. **SolanaIntegration** - For Solana blockchain
5. **TonIntegration** - For TON blockchain
6. **TronIntegration** - For Tron blockchain
7. **BlockchainIntegration** (abstract base class) - Base for blockchain integrations

## Database Consistency Rules

### Institutions that MUST have `has_integration = true`:

1. **Binance** (crypto_exchange) - Uses BinanceIntegration
   - Website: https://www.binance.com
   - Status: ✅ Fixed - Updated in Supabase on 2025-12-15

2. **All blockchain institutions** (crypto_wallet type):
   - Bitcoin, Ethereum, BSC, Polygon, Solana, Tron, TON, etc.
   - These use blockchain-specific integrations (Bitcoin, EVM, Solana, Ton, Tron)
   - Status: ✅ Marked by migration `0015_mark_blockchain_integrations.sql`

### Institutions that should have `has_integration = false`:

1. **Banks** (bank type) - No programmatic integration implemented
2. **Brokers** (broker type) - No programmatic integration implemented  
3. **Other crypto exchanges** (crypto_exchange type) - Only Binance has integration currently
   - Coinbase, Kraken, Bitfinex, etc. should remain `false` until integrations are implemented

## How to Add New Integrations

When adding a new institution integration:

### Step 1: Implement the Integration
Create the integration class in `packages/integrations/src/implementations/`

### Step 2: Update Database
Choose the appropriate method based on the type of change:

**Option A: Use Supabase MCP** (Recommended for simple data updates)
```typescript
// Via Supabase MCP execute_sql
UPDATE institutions 
SET has_integration = true 
WHERE website = 'https://institution-website.com';
```

**Option B: Create Migration via Drizzle CLI** (For schema changes or complex updates)
```bash
# 1. Update schema if needed
cd packages/core
bun run db:generate  # Generates migration from schema changes

# 2. Migrations are auto-generated by Drizzle based on schema.ts changes
# Do not manually create SQL files - let Drizzle generate them
```

### Step 3: Verify Consistency
Run this query to verify:
```sql
SELECT i.name, i.website, it.code as type, i.has_integration
FROM institutions i
JOIN institution_types it ON i.type_id = it.id
WHERE i.has_integration = true
ORDER BY it.code, i.name;
```

## Validation Query

Use this to verify all institutions with integrations match the implementations:

```sql
-- Count institutions by type with integrations
SELECT it.code, it.name, COUNT(i.id) as count
FROM institutions i
JOIN institution_types it ON i.type_id = it.id
WHERE i.has_integration = true
GROUP BY it.code, it.name
ORDER BY it.code;

-- Expected results:
-- crypto_exchange: 1 (Binance)
-- crypto_wallet: 43+ (all blockchain institutions)
```
