# Contributing to Scani Thank you for contributing to Scani! This document provides guidelines and instructions for contributing to the project. ## 🚀 Quick Start for New Contributors ### Prerequisites - [Bun](https://bun.sh/) (latest version) - PostgreSQL database (Supabase or local) - Node.js 18+ (for compatibility) ### Setup 1. **Clone and Install** ```bash git clone <repository-url> cd scani bun install ``` 2. **Environment Setup** ```bash # Backend cp apps/backend/.env.example apps/backend/.env.local # Edit with your values # Frontend cp apps/frontendV2/.env.example apps/frontendV2/.env.local # Edit with your values ``` 3. **Database Setup** ```bash # Apply migrations cd packages/core bun run db:migrate ``` 4. **Start Development** ```bash # Start all servers bun dev # Or start individually bun dev:backend bun dev:frontend bun dev:landing ``` ## 📋 Development Workflow ### 1. Create a Feature Branch ```bash git checkout -b feature/your-feature-name # or git checkout -b fix/bug-description # or git checkout -b docs/documentation-update ``` ### 2. Make Your Changes - Follow **Clean Architecture** principles (Use Cases → Services → Repositories) - Use **TypeScript** strictly (no `any` types) - Use **Decimal.js** for all financial calculations - Use **Drizzle ORM** for database operations (never raw SQL) - Use **TypeDI Container** for dependency injection ### 3. Code Quality Checks **Before committing, ensure all checks pass:** ```bash # Run linter bun lint # Auto-fix issues bun lint:fix # Run type checker bun type-check ``` ### 4. Test Your Changes - Manually test functionality in the UI - Verify backend API calls work correctly - Check WebSocket real-time updates if applicable - Test with different data scenarios ### 5. Commit Your Changes ```bash git add . git commit -m "feat: add new feature description" # or git commit -m "fix: resolve issue with holdings calculation" # or git commit -m "docs: update README with setup instructions" ``` **Commit Message Format:** - `feat:` - New features - `fix:` - Bug fixes - `docs:` - Documentation changes - `refactor:` - Code refactoring (no functional changes) - `chore:` - Maintenance tasks - `test:` - Adding or updating tests ### 6. Submit a Pull Request - Push to your feature branch - Create a PR with a clear description - Link any related issues - Wait for review ## 🏛️ Architecture Guidelines ### Clean Architecture Scani follows clean architecture with clear separation of concerns: ``` Presentation Layer (tRPC Routers) ↓ Application Layer (Use Cases) ↓ Infrastructure Layer (Services, Repositories) ↓ Domain Layer (Entities) ``` **Key Principles:** - **Routers** should be thin controllers (delegate to use cases) - **Use Cases** encapsulate business logic (single responsibility) - **Services** handle infrastructure (external APIs, complex calculations) - **Repositories** manage data access (Drizzle ORM operations) ### Dependency Injection All services use **TypeDI** for dependency injection: ```typescript import { Service } from 'typedi'; @Service() export class MyService { constructor( private readonly pricingService: PricingService, ) {} } // Retrieve from container const service = Container.get(MyService); ``` ### Import Guidelines **✅ CORRECT:** ```typescript import { MyService } from './MyService'; import { Container } from 'typedi'; const service = Container.get(MyService); ``` **❌ WRONG:** ```typescript const { MyService } = require('./MyService'); // No require() const service = new MyService(); // No direct instantiation const { MyService } = await import('./MyService'); // No dynamic imports ``` ## 🔧 Common Tasks ### Adding a New Feature 1. Create use case in `packages/core/src/use-cases/` 2. Add repository methods if needed in `packages/core/src/repositories/` 3. Create tRPC router endpoint in `apps/backend/src/presentation/routers/` 4. Add frontend components in `apps/frontendV2/src/components/` 5. Update types in `packages/shared/src/types/` ### Fixing a Bug 1. Identify the layer where the bug exists 2. Write a test case (if applicable) 3. Fix the issue following architecture patterns 4. Run `bun lint` and `bun type-check` 5. Test manually ### Adding Database Fields 1. Update schema in `packages/core/src/database/schema.ts` 2. Generate migration: `cd packages/core && bun run db:generate` 3. Apply migration: `bun run db:migrate` 4. Update repositories and use cases as needed ### Adding External API Integration 1. Create provider class with rate limiting 2. Add service in `packages/core/src/services/` 3. Register in TypeDI container 4. Use in use cases ## 📝 Code Style ### TypeScript - Use strict mode (no `any` types) - Use proper type annotations - Use Zod for runtime validation - Use Decimal.js for financial calculations ### Naming Conventions - **Files:** PascalCase for components, camelCase for utilities - **Classes:** PascalCase - **Functions/Variables:** camelCase - **Constants:** UPPER_SNAKE_CASE - **Types/Interfaces:** PascalCase ### Comments - Use JSDoc for public APIs - Explain _why_, not _what_ - Remove TODO comments when completed - Keep inline comments minimal ## 🧪 Testing ### Running Tests ```bash # Backend tests cd apps/backend bun test # Core package tests cd packages/core bun test ``` ### Writing Tests - Test use cases in isolation - Test routers with mocked dependencies - Test services with mocked external APIs - Use Decimal.js for financial assertions ## 🐛 Reporting Issues ### Bug Reports Include: - Description of the issue - Steps to reproduce - Expected vs actual behavior - Environment details (OS, Bun version, browser) - Screenshots if applicable ### Feature Requests Include: - Problem statement - Proposed solution - Use cases and examples - Alternative solutions considered ## 🔐 Security Guidelines - **Never** commit secrets or `.env` files - **Never** log sensitive data (passwords, tokens, keys) - **Always** validate user inputs with Zod schemas - **Always** scope queries to authenticated user - **Always** use parameterized queries (Drizzle ORM) - Report security vulnerabilities privately ## 📚 Resources - [README.md](./README.md) - Project overview and setup - [ARCHITECTURE.md](./docs/ARCHITECTURE.md) - Technical architecture details - [EXECUTIVE_SUMMARY.md](./docs/EXECUTIVE_SUMMARY.md) - Project status - [ROADMAP.md](./docs/ROADMAP.md) - Development roadmap - [Copilot Instructions](./.github/copilot-instructions.md) - Agent guidelines ## 💬 Questions? For questions or discussions: - Open an issue on GitHub - Check existing documentation in `/docs` - Review the architecture guide - Ask in the project's communication channel --- **Thank you for contributing to Scani!** 🎉
